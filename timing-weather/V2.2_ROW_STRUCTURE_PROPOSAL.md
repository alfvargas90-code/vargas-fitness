# V2.2 — Row Structure Proposal (Timing Weather)

**Status:** PROPOSAL — awaiting Alfie's approval. No files changed.
**Why:** v2.1 is 14 flat `<section>`s in one shell. A 3-col auto-flow grid over flat
sections produces a messy approximation because the browser packs cards in DOM order
with no notion of which cards belong together or how wide each should be. The mockup is a
*curated composition* (some cards span 2 cols, one rail spans 2 rows). We must encode that
grouping in HTML **before** CSS, or CSS can only guess.

---

## 2. Proposed row containers

- **Row 0 (full-bleed):** Hero Solar Intelligence [§1]
  *— standalone showpiece; owns the fold.*
- **Row 1 (full-bleed strip):** Now Bar [§2]
  *— at-a-glance "right now" snapshot; a divider between hero and detail.*
- **Row 2 (curated 3-col grid; right rail spans 2 sub-rows / 1-col on mobile):**
  - `reading` → Daily Reading [§3]
  - `changed` → What Changed [§4]
  - `upcoming` → Upcoming Conditions [§7] *(spans cols 1–2, the lower sub-row)*
  - `rail` → Today's Insight [§5] + Recommended Actions [§6] *(col 3, spans both sub-rows)*
  *— the "what's happening / what changed / what to do & what's coming" cluster. The right
  rail is the persistent advice column the eye returns to; left two cols carry state + delta +
  horizon. This is the single block that auto-flow cannot reproduce — it needs named areas.*
- **Row 3 (4-col desktop / 2-col tablet / 1-col mobile):**
  Sky Conditions [§9] | Planet Influences [§10] | Top Drivers [§11] | Why This Forecast [§12]
  *— the "evidence / drill-down" cluster: the four explain-the-score cards, equal weight.*
- **Row 4 (2-col desktop / 1-col mobile):** Current Phase [§8] | Confidence [§13]
  *— low-priority "meta / trust" footer cards; small, side by side, out of the main flow.*
- **Row 5 (full-bleed):** Footer Nav [§14]

## 3. Mobile rule

Every row collapses to **a single column, preserving DOM order** — no reordering needed
because DOM order is authored to read correctly on mobile:

`Hero → Now Bar → Daily Reading → What Changed → Today's Insight → Recommended Actions →
Upcoming Conditions → Sky → Planet → Drivers → Why → Current Phase → Confidence → Footer`

The only desktop-vs-mobile difference is positional, handled by CSS, not DOM: on desktop the
`grid-template-areas` in Row 2 pull **Upcoming Conditions** up beside the rail and let the
rail span two sub-rows; on mobile those areas dissolve and the cards stack in plain DOM order
(Insight + Actions land right after What Changed, exactly the mobile reading priority we want).

## 4. HTML skeleton (wrappers only — existing `<section>`s untouched inside)

```html
<div class="timing-weather-shell">

  <!-- Row 0 -->
  <div class="row row--bleed">
    <section id="section-1-hero" …>…</section>
  </div>

  <!-- Row 1 -->
  <div class="row row--bleed row--strip">
    <section id="section-2-now-bar" …>…</section>
  </div>

  <!-- Row 2 — curated composition (grid-template-areas) -->
  <div class="row row--today">
    <section id="section-3-daily-reading"     style="grid-area:reading"  …>…</section>
    <section id="section-4-what-changed"       style="grid-area:changed"  …>…</section>
    <div class="rail" style="grid-area:rail">
      <section id="section-5-todays-insight"   …>…</section>
      <section id="section-6-recommended"      …>…</section>
    </div>
    <section id="section-7-upcoming-conditions" style="grid-area:upcoming" …>…</section>
  </div>

  <!-- Row 3 — analytics -->
  <div class="row row--4col">
    <section id="section-9-sky-conditions"    …>…</section>
    <section id="section-10-planet-influences" …>…</section>
    <section id="section-11-top-drivers"       …>…</section>
    <section id="section-12-why-forecast"      …>…</section>
  </div>

  <!-- Row 4 — meta -->
  <div class="row row--2col">
    <section id="section-8-current-phase" …>…</section>
    <section id="section-13-confidence"   …>…</section>
  </div>

  <!-- Row 5 -->
  <nav id="section-14-footer-nav" class="row row--bleed footer-nav" …>…</nav>

</div>
```

*(`grid-area` shown inline for clarity; in the build they move to the stylesheet. Row 2's
container declares: `grid-template-areas:"reading changed rail" "upcoming upcoming rail";`)*

---

## 5. What this UNBLOCKS for CSS

Each `.row--*` becomes its **own grid context** with an explicit column count, so CSS sets
spans deliberately instead of hoping auto-flow packs 14 cards correctly. Row 2's
`grid-template-areas` makes the curated mockup layout — wide Upcoming, tall right rail —
deterministic and exactly matchable, no auto-placement randomness. And mobile is a one-line
collapse per row (`grid-template-columns:1fr; grid-template-areas:none;`) because DOM order is
already the correct mobile order.
